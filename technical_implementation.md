# 亚马逊智能客服 Agent (MVP v1.0) 技术实现报告与方案

本报告基于系统代码库真实现状与《Agent F 亚马逊智能客服 Agent (MVP v1.0) 需求文档》对比分析生成，明确了当前进度、差距以及下一步开发的技术实现路径。

---

## 一、 目前已实现功能与现状

当前项目（`smart-cs`）处于极早期的核心链路验证阶段（PoC阶段），主要验证了 SpringBoot 环境下使用 LangChain4j 接入国产大模型的可行性：

1. **基础框架骨架搭建**
   - 初始化了标准 Java / Spring Boot 3.x 结构。
2. **大模型底座与 AI 生成接入**
   - 实现了集成 `LangChain4j` 并通过 OpenAI 兼容模式连接阿里云通义千问大模型 (`qwen-plus`)。
   - 编写了基础业务提示词：在 `QwenAiService` 中实现将模拟的买家抱怨转换为具有同理心（Empathy）、提供补发（Replacement）方案的专业英文回复。
   - 提供了 `/api/test-ai` 端点供开发者直接验证生成效果。
3. **MOCK 接口测试**
   - 提供了 `/mock/messages` 回调，用于模拟前端或外部推送单条买家文本数据。

## 二、 需求差距分析（未实现功能清单）

> [!WARNING]
> 对照需求文档标准 MVP v1.0，当前项目数据库、SP-API 接口、前端系统、RAG系统、高可用与安全合规模块均**尚未开发**。

### 1. 核心业务层 (亚马逊对接 & RAG)
- ❌ **亚马逊(SP-API)集成缺失**：未实现 OAuth 授权管理；未接入 Amazon SQS 事件监听机制拉取真实消息；不存在订单查询和回复发送端点实现。
- ❌ **RAG 知识库管理与集成缺失**：没有任何上传、解析（PDF/OCR）、文件切块及向量化入库的后端处理接口；生成草稿时未携带相关向量检索上下文。
- ❌ **AI 辅助深度不足**：缺少异步流水线对新消息进行意图识别、提取摘要和合规词校验预警；未记录低置信度的兜底修正数据闭环。

### 2. 用户与系统基础层
- ❌ **数据库存储未配置**：未配置 PostgreSQL 和 pgvector，无实体类映射 (ORM)。
- ❌ **用户分级与限额验证**：缺失针对 Starter 和 Pro 套餐的每月调用次数计算与阻断逻辑；无鉴权模块。
- ❌ **前端工作台视图**：目前为纯后端，尚未提供任何 PC 端响应式网页。
- ❌ **实时通知通道**：未搭建确保桌面通知（<5 秒）的 WebSocket 或轮询通道。

### 3. 非业务功能控制 (非功能性要求)
- ❌ **API 并发与流控体系**：未实现利用 Redis 实现的防封限流（令牌桶），以及单店铺并发隔离队列。
- ❌ **安全隐私隔离**：缺失 AES-256 加解密存储、30天 PII 数据硬删除任务调度以及按 `seller_id` 的租户数据隔离机制。

## 三、 下一步技术实现文档 (Technical Implementation Path)

为此项目的规模与技术堆栈，建议后续遵照以下模块化的技术方案逐步落地：

### 阶段 1：基础设施与基座闭环结构构建
1. **数据库初始化设计**：
   - Docker 部署带 `pgvector` 扩展的 PostgreSQL 实例。
   - 引入 `Spring Data JPA`，设计多租户模型实体（Users, Stores, Messages, Documents, Vectors）。设计软删除和定期擦除调度 (`@Scheduled`) 以达到 30天销毁合规要求。
2. **多租户数据防护与高可用配置**：
   - 引入 Redis 以使用 `Redisson` 实现基于分布式令牌桶的 API 调用限流，规避亚马逊 API 封禁风险。
   - Spring Security集成，实现租户级别 JWT 登录与权限管理。

### 阶段 2：亚马逊 SP-API 实战对接
3. **SP-API 监听系统与队列处理**：
   - 集成 AWS SDK (Java V2)，创建 SQS 消费者服务，采用隔离队列池分发（每个 `seller_id` 进行动态调度分配），确保单一店铺消费不会堵塞整体消息链路。
   - 针对不同亚马逊区域，设置 OAuth Token（AES-256存取）缓存并具备自动轮换机制。
4. **单体流程解耦**：
   - 把目前的 `QwenAiService` 重构进一个异步流转服务中（新消息 -> 意图识别/翻译 -> 检索知识库 -> AI 草稿生成）。

### 阶段 3：RAG 向量工程落地
5. **知识库文档处理流**：
   - 开发文件上传 OSS 服务，后台集成 Python 或外部 OCR API 进行 `pdfplumber` 兼容分析；提取纯文本后，利用 `text-embedding-v2` 把数据切片、存入 `pgvector` 库。
   - AI回答时，通过 Langchain4j 提取向量数据库近邻词块绑定 Prompt 补充给生成大模型。

### 阶段 4：前端工作台及双向通信
6. **前后端交互与界面落地**：
   - 引入 Spring WebSocket (`@EnableWebSocketMessageBroker`) 和 STOMP 协议，实现当处理队列有新解析生成的 AI 草稿时，即时下发至 Web 客户端。
   - 前端采用 React/Next.js 等现代化栈实现响应式开发并对接真实 API 端点。
